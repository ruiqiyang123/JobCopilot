# JobCopilot 四阶段改造设计

日期：2026-07-14  
状态：用户已批准并要求实施

## 目标与顺序

在现有多模型分支上依次完成四个可独立验收的阶段：

1. 补齐内置 LongCat 服务商。
2. 增加可配置的经验和公司规模硬筛选。
3. 增加默认模拟运行、二次校验和失败即停。
4. 增加插件处理岗位的本地进度看板。

每一阶段必须在自动测试、语法检查、密钥扫描通过并推送 GitHub 后才能标记完成并进入下一阶段。测试和验收不执行真实 BOSS 沟通或投递。

## 第一阶段补充：LongCat

LongCat 作为内置 OpenAI Chat Completions 兼容服务商：

```text
provider: longcat
baseUrl: https://api.longcat.chat/openai/v1
model: LongCat-2.0
authType: bearer
tokenParameter: max_tokens
jsonMode: false
```

扩展为 `https://api.longcat.chat/*` 申请固定主机权限。API Key 只保存在 `chrome.storage.local`，不进入源码、测试、普通日志或 Git 提交。

## 第二阶段：可配置硬筛选

经验和公司规模各自具有“启用”开关和分类多选。启用时至少选择一项；关闭时忽略该维度。

- 经验档位：经验不限、应届/在校、1 年以内、1–3 年、3–5 年、5–10 年、10 年以上。
- 公司档位：0–20、20–99、100–499、500–999、1000–9999、10000 人以上。
- 默认经验：1 年以内、1–3 年。
- 默认公司：所有 100 人以上档位。

岗位卡片采集新增 `experience` 和 `companySize`。纯函数筛选器返回：

```javascript
{
  filterStatus: 'pass' | 'fail' | 'pending',
  filterReasons: string[]
}
```

已知不匹配为 `fail`；没有已知不匹配但缺失必要信息为 `pending`；其余为 `pass`。硬筛选发生在 AI 调用前。`pending` 不自动 AI 筛选、不默认勾选、不自动投递。用户只可对 `pending` 做人工确认；人工确认不能覆盖已知的明确不匹配。

## 第三阶段：模拟运行与投递安全

运行模式分为 `preview` 和 `live`，每次打开侧边栏都默认 `preview`，不保存正式模式。

完整预演依次执行：收集、硬筛选、AI 筛选、读取完整 JD、岗位身份与筛选条件二次校验、生成招呼语并展示。预演不得调用建立沟通或发送消息的代码路径。

正式投递只接受预演成功的岗位。用户需进行一次批次确认；发送前再次读取当前岗位并校验 ID、名称、公司、筛选条件、人工确认和已投记录。正式投递复用用户在预演中看到的招呼语。任何岗位发生校验、模型、页面或发送失败时立即停止整个批次，保存成功、失败和未执行列表，不自动重试。

## 第四阶段：进度管理

第一版只纳入插件收集或投递的岗位，不扫描 BOSS 历史。岗位状态为：

```text
collected  已收集
contacted  已沟通
replied    已回复
interview  面试
rejected   淘汰
offer      Offer
```

岗位收集时建立或更新记录；正式发送成功后自动转为 `contacted`；其余状态由用户手动选择。记录保存岗位基础信息、当前状态、更新时间和状态历史，使用岗位 ID 去重并保存在 `chrome.storage.local`。

## 安全与验收边界

- 自动化测试使用纯函数、Mock fetch 和静态集成断言。
- 手工验收只允许连接模型和模拟运行，不执行真实沟通或发送。
- 所有动态 HTML 均经过转义。
- API Key 不出现在仓库、错误明文或测试快照中。
- 未通过上一阶段验收时不开始下一阶段。
