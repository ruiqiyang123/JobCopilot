# JobCopilot 城市识别、快速 AI 筛选与批量批准设计规格

日期：2026-07-15  
状态：用户已批准

## 1. 背景

当前审核流程暴露出三个相互关联的问题：

1. 城市与行政区解析会扫描岗位标题和完整 JD，并使用过宽的中文正则，因此可能把“深圳互联网证券产品经理”截断成城市，把“活动资讯社区”识别成行政区，最终错误排除目标城市岗位。
2. “确认信息符合”会同步等待一次完整六维 AI 评分。按钮只进入禁用状态，没有持久化的处理中反馈；LongCat 返回较慢或输出无效 JSON 时，用户会长时间感觉没有反应。
3. 六维评分在每个岗位上自动生成，耗时和模型消耗较高，但用户主要需要快速判断岗位是否值得审核；推荐岗位仍需逐个点击批准。

## 2. 目标

- 可靠识别明确的页面位置，禁止从岗位标题、公司名或完整 JD 猜测城市和行政区。
- 当页面没有可靠城市信息时，以本批搜索城市作为兜底并通过城市筛选。
- 将批量 AI 处理改为短输出的快速筛选，将六维详细评分改为按需生成。
- “确认信息符合”立即保存人工确认并显示可恢复的 AI 处理中状态，不阻塞其他岗位操作。
- 支持选择推荐岗位并一键批准；批量批准只改变审核状态，不直接发起 BOSS 沟通。

## 3. 非目标

- 不修改审核分类标签布局，不增加固定的岗位数量汇总，也不调整筛选完成后默认打开的分类。
- 不允许位置兜底覆盖页面明确显示的异地城市。
- 不让待补充、硬筛选排除或 AI 排除的岗位默认进入批量批准范围。
- 不取消投递预演、预演确认、岗位身份复核和正式投递批次确认。
- 不保证模型网络请求即时完成；本次目标是减少自动请求的输出规模，并让异步状态明确、可恢复。

## 4. 城市与行政区识别

### 4.1 可信来源

位置解析只接受以下来源：

1. 搜索卡片或岗位详情页中独立的位置 DOM 字段。
2. 独立标签中符合城市白名单或“城市 + 行政区”结构的文本。
3. 用户人工确认的位置。
4. 页面没有可靠位置时的本批搜索城市。

岗位标题、公司名称和完整 JD 仍可用于岗位语义匹配，但不得作为位置证据。包含“社区”“园区”“业务区”“区域”等普通 JD 词语的句子不得被解析为行政区。

### 4.2 优先级与冲突

岗位保存 `citySource`：

- `page`：页面可靠位置字段。
- `manual`：用户人工确认。
- `search`：使用本批搜索城市兜底。

优先级为 `manual > page > search`。页面明确城市与目标城市不一致时，硬筛选正常排除；页面位置缺失或不可靠时，写入搜索城市、`citySource: search`，并通过城市筛选。

行政区没有可靠证据时保持为空，不从搜索城市推断。若用户启用了行政区硬筛选，行政区缺失仍按现有规则进入待补充。

### 4.3 旧批次迁移

新增位置解析版本。恢复旧批次时重新处理没有可信来源或由旧解析器生成的位置字段：

- 从独立位置事实重新解析。
- 无可靠城市时使用当前搜索城市兜底。
- 清除“活动资讯社区”等不满足新规则的旧行政区。
- 重新执行硬筛选，使被旧解析器错误排除的岗位回到正确分类。

审核卡片对 `citySource: search` 显示“城市：深圳（按搜索条件推定）”，避免把兜底值伪装成页面精确信息。

## 5. 两级 AI 筛选

### 5.1 自动快速筛选

硬筛选通过后，批量流程只请求以下短结构：

```json
{
  "decision": "recommended | needs_info | excluded",
  "reason": "一句基于 JD 和简历事实的理由"
}
```

快速筛选继续使用完整 JD 和简历事实，但不自动生成六个维度、0–100 总分、优势列表和风险列表。请求使用较低的输出 token 上限与低温度，降低等待和消耗。

解析器兼容纯 JSON 和 Markdown 代码块；字段非法时不默认排除，岗位进入“待确认”并显示“重新快速判断”。同一岗位正在请求时禁止重复提交。

### 5.2 按需详细评分

六维评分保留为卡片操作“生成详细评分”。用户点击后才调用现有详细评分能力，结果独立缓存到岗位记录中。

详细评分只提供补充判断信息，不改变已经完成的快速筛选决策，不阻止批准岗位，也不参与批量批准资格计算。失败时只显示详细评分错误和重试按钮。

## 6. “确认信息符合”异步流程

点击后按以下顺序处理：

1. 立即将硬筛选缺失项记录为人工确认，设置 `manualOverride: true`。
2. 持久化岗位，按钮立即显示绿色“✓ 信息已确认”。
3. 设置 `aiScreeningStatus: running`，卡片显示“AI 快速判断中…”，用户可以继续审核其他岗位。
4. 后台完成快速筛选后写入决策与理由，将岗位移动到对应现有分类。
5. 请求失败或超时时设置 `aiScreeningStatus: failed`，保留人工确认，显示“重新快速判断”；用户不需要再次确认缺失信息。

侧边栏关闭或 Service Worker 恢复时，已确认事实不得丢失。恢复到长期停留在 `running` 的旧任务时，将其改为可重试的 `failed`，不自动重复消耗模型额度。

## 7. 批量批准

### 7.1 资格

批量批准只包括同时满足以下条件的当前批次岗位：

- `reviewStatus: pending_review`。
- 硬筛选通过，或信息缺失已经有明确人工确认记录。
- 快速 AI 决策为 `recommended`。
- 未成功投递、未在其他审核操作中处理。

待补充、AI 判断中、AI 排除、硬筛选排除、不投递和已批准岗位不进入默认选择。

### 7.2 交互

- 推荐岗位卡片增加复选框。
- 推荐分类提供“全选推荐岗位”和绿色按钮“一键批准推荐岗位（N）”。
- 用户可取消个别复选框后批准剩余岗位。
- 点击后直接批量写入 `reviewStatus: approved`，不弹二次确认；操作结果显示“已批准 N 个岗位”。
- 批量批准不是正式投递。岗位仍需完成投递预演、逐岗预演确认以及正式投递批次确认。
- 用户可以在现有“已批准”分类逐个取消批准。

批量更新必须一次性持久化并只触发一次侧边栏刷新，避免 N 个岗位产生 N 次存储和重绘。

## 8. 数据字段

岗位新增或明确使用以下字段：

```js
{
  city: "深圳",
  district: "南山区",
  citySource: "page | search | manual",
  locationParseVersion: 2,
  aiScreeningStatus: "idle | running | succeeded | failed",
  aiScreeningError: "",
  quickDecision: "recommended | needs_info | excluded",
  quickReason: "",
  detailedScore: null,
  detailedScoreStatus: "idle | running | succeeded | failed"
}
```

现有 `matchDecision`、`reason` 和详细评分字段在迁移期继续兼容。读取旧数据时优先采用新字段；写入新结果时同步必要的兼容字段，避免破坏现有审核工作流。

## 9. 错误处理与安全边界

- 明确异地岗位仍由城市硬筛选排除。
- 搜索城市兜底必须带 `citySource: search`，不得覆盖 `page` 或 `manual`。
- 快速 AI 格式错误、超时或网络失败只进入待确认，不自动批准、不自动投递。
- 批量批准只处理服务端/后台再次校验后仍符合资格的岗位；前端复选框不是唯一安全判断。
- 正式投递继续逐岗校验岗位 ID、预演有效性、重复投递与页面身份。

## 10. 测试与验收

### 10.1 位置解析

- “深圳互联网证券产品经理”不会被截成“圳互联网证券产品经理”。
- “活动资讯社区”不会被识别为行政区。
- 独立位置“深圳·南山区”解析为页面来源。
- 页面缺少位置且搜索城市为深圳时，岗位得到 `深圳 + citySource: search` 并通过城市筛选。
- 页面明确为广州时，即使搜索城市为深圳也必须排除。
- 旧批次的错误位置在迁移后重新筛选。

### 10.2 快速 AI 与异步确认

- 批量筛选不再自动调用六维详细评分。
- 快速 JSON、代码块 JSON 和非法返回均有确定结果。
- 点击“确认信息符合”后先持久化人工确认，再开始 AI 请求。
- 处理中不能重复提交；成功、失败和恢复状态正确。
- 详细评分只在用户点击后请求，失败不影响批准。

### 10.3 批量批准

- 只有符合资格的推荐岗位可以被选择。
- 全选、取消个别选择和批量批准数量正确。
- 后台对岗位资格二次校验。
- 批量操作一次持久化，刷新后状态保留。
- 批量批准不会调用预演或正式投递接口。

## 11. 验收标准

- 截图中的平安银行岗位不会再因伪造城市而被排除。
- 位置缺失的深圳搜索结果默认按深圳通过城市条件。
- “确认信息符合”点击后 300 毫秒内出现明确的已确认与处理中反馈。
- 自动 AI 请求只产生快速判断，详细评分默认不生成。
- 用户可以选择推荐岗位并通过一个批量操作全部批准。
- 现有投递安全门和岗位进度数据保持不变。
