# JobCopilot 预演进度反馈修复规格

日期：2026-07-14

## 背景与问题

用户批准岗位并点击“预演已批准岗位”后，界面仍显示“等待预演”，没有可见反馈。

现有侧边栏会先进入运行态，再使用不等待结果的 `chrome.runtime.sendMessage` 启动预演。后台立即回复“已接收”，但初始化校验、读取岗位详情、二次校验和生成 AI 开场都在后续异步任务中执行。若这些步骤失败，错误只进入默认折叠的运行日志；后台也不会为初始化错误发送可供预演列表渲染的结果。因此岗位卡片继续显示“等待预演”，用户会认为按钮没有生效。

## 目标

1. 点击预演后立即显示明确的批次运行状态。
2. 在每个已批准岗位卡片内显示当前预演步骤。
3. 所有失败都直接显示在对应岗位或批次区域，不再依赖折叠日志。
4. 保持“任一岗位失败立即停止整个批次”的安全规则。
5. 不改变审批、三段式招呼内容和正式投递逻辑。

## 已确认交互方案

采用“卡片内实时进度”方案。

点击“预演已批准岗位”后：

- 按钮立即变为“预演中 0/N”并禁用，避免重复启动。
- 所有待执行岗位立即显示“排队等待”。
- 当前岗位按顺序显示：
  1. 读取完整 JD
  2. 二次校验岗位
  3. 生成 AI 个性化开场
  4. 等待用户确认
- 成功生成后，卡片展示三段式预演内容和“确认此岗位预演”按钮。
- 当前岗位失败时，卡片直接显示具体失败原因。
- 因前一个岗位失败而没有执行的岗位显示“本批次未执行”。
- 批次结束或失败后，预演按钮恢复可用。

## 状态与消息设计

### 前端临时状态

侧边栏维护独立的 `previewRunState`，用于展示本次运行过程，不写入已冻结的预演快照：

- `runId`
- `status`: `idle | running | completed | failed | stopped`
- `completed`
- `total`
- `jobs[jobId].stage`
- `jobs[jobId].error`

岗位阶段包括：

- `queued`
- `reading_detail`
- `verifying`
- `generating_opening`
- `draft`
- `failed`
- `not_run`

最终生成的预演仍使用现有 `sw_previews` 持久化结构。运行进度和已确认预演必须保持分离，避免临时状态误开正式投递安全门。

### 后台消息

`START_PREVIEW` 在完成批次前置校验后返回启动确认，包含 `runId` 和实际执行岗位 ID。若招呼方案、岗位集合或配置无效，直接返回 `ok: false` 和可读错误。

后台在每个阶段广播 `PREVIEW_PROGRESS`：

- `runId`
- `jobId`
- `stage`
- `completed`
- `total`
- `error`（仅失败时）

批次结束广播现有 `PREVIEWED`，并包含最终 `lastBatch`、`previews` 和岗位状态。批次级初始化异常也必须产生失败结果，不能只写日志。

## 数据流

1. 侧边栏验证当前招呼方案并收集已批准岗位 ID。
2. 侧边栏先渲染全部岗位为 `queued`，再等待 `START_PREVIEW` 的启动确认。
3. 后台完成配置、方案和岗位前置校验，创建唯一 `runId`。
4. 后台逐个岗位读取详情、校验并生成开场，在每一步广播进度。
5. 成功岗位写入草稿预演并显示待确认内容。
6. 失败岗位写入失败预演，剩余岗位标记为 `not_run`，随后结束批次。
7. 侧边栏收到结束消息后清理运行态、恢复按钮并渲染最终结果。

## 错误处理

- 启动失败：恢复按钮，在预演区域顶部显示批次错误，岗位不再永久停留在“等待预演”。
- 详情读取失败：对应岗位显示具体原因，剩余岗位显示“本批次未执行”。
- 模型调用失败：对应岗位显示模型错误，停止批次。
- 消息通道断开：侧边栏显示“扩展后台未响应，请在扩展页刷新后重试”。
- 侧边栏被关闭后重开：从 `lastBatch` 和持久化预演恢复最终状态；不把中断的运行误认为成功。
- 所有错误继续写入运行日志；出现错误时自动展开日志，但卡片错误是主要反馈。

## 安全边界

- 预演路径不得调用建立沟通、发送文字或上传图片的消息。
- 只有 `draft` 预演经用户逐岗位确认后才能成为 `confirmed`。
- 临时运行状态不得参与正式投递资格判断。
- 正式投递仍要求岗位已批准、预演已确认且指纹未过期。
- 本修复不改变正式投递发送顺序和失败即停规则。

## 测试与验收

### 自动测试

- 点击预演后，已批准岗位立即进入 `queued`。
- 后台按 `reading_detail → verifying → generating_opening → draft` 顺序发出进度。
- 前置校验失败会返回可见错误并恢复按钮。
- 当前岗位失败会显示错误，剩余岗位全部变为 `not_run`。
- 成功岗位仍可编辑并确认 AI 开场。
- `PREVIEW_PROGRESS` 不会创建已确认预演。
- 预演代码路径不含建立沟通、发送文字或上传图片。

### 本地验收

- 全量 Node 测试、JavaScript 语法检查、`git diff --check` 和 API Key 扫描通过。
- 使用 Mock 岗位完成一次成功批次和一次失败批次。
- 用户刷新本地 Chrome 扩展后，用少量真实岗位验证预演反馈。
- 未获得单独授权前不执行真实正式投递。

## 非目标

- 不修改岗位筛选标准。
- 不修改招呼方案的三段内容。
- 不改变正式投递资格和发送实现。
- 不增加自动重试。
- 不扫描 BOSS 历史沟通记录。

## 完成标准

点击“预演已批准岗位”后，用户在一个界面刷新周期内看到按钮运行态和岗位排队状态；预演过程中的每个岗位始终具有明确的进行中、成功、失败或未执行状态，不再出现无反馈的永久“等待预演”。
