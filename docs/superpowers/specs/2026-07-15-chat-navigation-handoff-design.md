# JobCopilot 聊天页跳转交接设计

日期：2026-07-15

## 背景

正式投递已能通过岗位、筛选和预演校验，但在点击“立即沟通/继续沟通”后会出现两类失败：

1. `建立沟通失败：未进入聊天页面`。
2. `A listener indicated an asynchronous response ... but the message channel closed`。

2026-07-15 的只读排查证实，第二类错误发生时，浏览器已于同一时刻访问对应岗位的 `/web/geek/chat?...jobId=...` 页面。详情页跳转会销毁内容脚本，内容脚本无法再通过原消息通道回传成功结果；后台却在观察实际页面前就把通道关闭当成建联失败。

另一个竞态来自 `waitTabComplete()`：发送点击命令后，原详情页本身已是 `complete`，因此该函数可能在新导航开始前就立即返回，随后的固定延时与 URL 检查会过早地判定“未进入聊天页面”。

## 目标

将建立沟通的成功条件从“详情页内容脚本成功回执”改为“浏览器实际进入与当前岗位匹配的 BOSS 聊天页”。

完成后需要：

- 支持详情页在同一标签页跳转。
- 支持 BOSS 新建标签页打开聊天。
- 详情页跳转导致消息通道关闭时，只有观察到匹配聊天页后才继续。
- 严格核对聊天 URL 中的 `jobId`，防止向其他岗位或 HR 发送内容。
- 真实未跳转、按钮不存在或岗位身份不一致时，保持失败立即停止。

## 已选方案

采用用户确认的方案 B：在发送 `GO_CHAT` 之前启动聊天页观察器，同时监听原标签页和点击后新建的 BOSS 标签页，以匹配岗位 ID 的聊天 URL 作为建联成功的唯一页面依据。

### 方案对比

#### 方案 A：忽略通道关闭并轮询原标签页

改动最小，但无法可靠处理 BOSS 在新标签页打开聊天的情况，也无法准确管理后续注入和标签页清理。

#### 方案 B：聊天页观察器（采用）

在点击前建立监听，同时覆盖同标签和新标签导航。页面事实与内容脚本回执分离，安全性和可测试性最好。

#### 方案 C：内容脚本点击前先回执

可避免首次点击销毁消息通道，但“继续沟通”可能是后出现的二次操作，仍需要跨页状态机；且无法单独解决新标签页问题。

## 组件与边界

### 1. 聊天 URL 规则模块

新增独立、无 Chrome API 依赖的规则模块，职责仅包含：

- 识别 URL 是否为 HTTPS BOSS 聊天页。
- 提取聊天 URL 中的 `jobId`。
- 将 URL 与预期岗位 ID 进行严格匹配。
- 识别因页面导航导致的瞬时消息通道错误。

该模块不读取页面、不点击按钮、不发送消息，可使用 Node 单元测试完整覆盖。

### 2. 后台聊天页观察器

后台新增一次性观察器，由正式投递流程创建。创建时记录：

- 原详情标签页 ID。
- 点击前已存在的标签页 ID 集合。
- 预期岗位 ID。
- 30 秒截止时间。

观察器同时使用 `chrome.tabs.onUpdated`、`chrome.tabs.onCreated` 和有限轮询检查以避免丢失早期事件。它记录新标签的 `openerTabId`，并只接受：

- 原详情标签页导航到匹配的聊天 URL；或
- 观察器创建后新出现、且 URL 中 `jobId` 匹配的 BOSS 聊天标签页。当 BOSS 保留 `openerTabId` 时，该值必须指向原详情页；当浏览器因 `noopener` 不保留来源时，仍必须同时满足“点击后新建”和“`jobId` 精确匹配”。

点击前已存在的其他聊天标签页不能被误用为当前岗位的目标页。

观察器在成功、失败、超时或批次中止后都必须移除监听器与定时器。

### 3. 内容脚本点击器

`content-search.js` 继续负责在已校验的详情页上查找并点击“立即沟通/继续沟通”，但它的回执不再作为建联成功的最终证据。

明确错误仍立即生效，例如“未找到立即沟通按钮”。若点击命令的消息通道在导航时关闭，后台不直接判失败，而是继续等待聊天页观察器。

## 正式投递数据流

1. 完成岗位身份、筛选条件、重复投递和预演指纹校验。
2. 记录当前标签页快照，启动与当前岗位 ID 绑定的聊天页观察器。
3. 向详情页发送 `GO_CHAT`。
4. 如果点击器返回明确错误，取消观察并立即停止批次。
5. 如果点击器成功回执，或因导航而关闭消息通道，继续等待观察器。
6. 观察器只在找到对应 `jobId` 的聊天页时返回目标标签页。
7. 等待该目标标签页加载完成，注入 `content-chat.js`。
8. 按已确认预演发送 AI 开场、固定消息和简历图片。
9. 三段都成功后才记录为“已沟通”和“已处理”。
10. 清理由本次投递创建的临时详情页与聊天页；不关闭用户原本已打开的标签页。

## 并发与竞态规则

- 正式投递仍严格串行，同一时刻只处理一个岗位。
- 聊天页观察器必须在发送 `GO_CHAT` 前启动，防止丢失立即导航事件。
- 内容脚本回执与聊天页观察结果独立，通道关闭不会自动变成成功，它只允许后台继续等待页面证据。
- 原详情页或明确以原详情页为 `openerTabId` 的新标签进入错误岗位时，立即报“岗位身份不一致”并停止。与原详情页无明确来源关系、且 `jobId` 不匹配的其他新标签必须忽略，不能注入或发送。
- 观察器超时后不允许自动重新点击，避免创建重复会话或无意重试。

## 错误处理

### 可继续等待的情况

- 详情页导航导致内容脚本消息通道关闭。
- `GO_CHAT` 命令成功回执，但聊天页仍在加载。

上述情况都必须在 30 秒内观察到与当前岗位匹配的聊天 URL，否则仍判定失败。

### 立即失败的情况

- 找不到“立即沟通/继续沟通”按钮。
- 标签页被关闭、BOSS 登录失效或脚本无法注入。
- 原详情页或其明确打开的新标签进入聊天路径，但 URL 缺少可核验的 `jobId`。
- 原详情页或其明确打开的新标签的聊天 `jobId` 与当前岗位不一致。
- 30 秒内未进入对应岗位聊天页。

用户可见错误要区分：

- `未找到立即沟通按钮`。
- `点击沟通后 30 秒内未进入对应岗位聊天页`。
- `聊天页岗位身份不一致，已停止发送`。
- `聊天页脚本注入失败`。

## 标签页归属与清理

- 从用户已打开的搜索页或详情页开始时，这些标签页始终属于用户，不自动关闭。
- 后台为完整 JD 校验新建的详情页属于当前投递任务，完成或失败后可关闭。
- 由该临时详情页点击后新建的聊天页同样属于当前投递任务，发送完成或失败后可关闭。
- 若聊天在原临时详情页内同标签跳转，只关闭该同一标签页一次。
- 清理前按标签页 ID 去重，忽略已被 BOSS 或用户关闭的标签页。

## 日志与可观测性

每个岗位的建联步骤增加结构化状态：

- `已点击沟通，等待对应岗位聊天页`。
- `详情页跳转中，已忽略瞬时消息通道关闭`。
- `已进入对应岗位聊天页`。
- 失败时记录失败阶段，但不记录完整 `securityId`、聊天 URL、招呼语或 API Key。

## 测试

### 单元测试

- 识别合法 BOSS 聊天 URL 并提取 `jobId`。
- 拒绝非 HTTPS、非 BOSS 域名、非聊天路径和缺少 `jobId` 的 URL。
- 接受岗位 ID 匹配的聊天 URL，拒绝错误岗位 ID。
- 识别 Chrome 因页面导航产生的消息通道关闭错误，不把其他脚本错误降级为可忽略。

### 后台流程测试

- 观察器在 `GO_CHAT` 之前启动。
- 原标签页导航到正确 `jobId` 时继续发送。
- 点击后新建标签页进入正确 `jobId` 时继发送。
- 消息通道关闭但观察到正确聊天页时继续。
- 消息通道关闭且未观察到聊天页时超时停止。
- 明确返回“未找到按钮”时立即停止。
- 原详情页或其子标签进入错误 `jobId` 时立即停止，不注入、不发送。
- 用户同时打开的无关聊天标签不会被当成当前岗位，也不会导致批次误停止。
- 只有找到匹配聊天页后才可出现 `SEND_BUNDLE`。
- 成功、错误、超时和批次中止后都清理监听器。
- 同标签与新标签的临时页清理不重复，不关闭用户旧标签页。

### 回归验证

- 全部现有单元测试通过。
- JavaScript 语法、Manifest JSON、差异格式和 API Key 扫描通过。
- 使用 Mock Chrome Tabs API 验收，未经单独授权不点击 BOSS 沟通按钮、不发送真实消息。

## 非目标

- 不修改岗位筛选、人工批准、预演确认和三段消息内容。
- 不取消发送前岗位身份、资格和重复投递校验。
- 不在本次同时调整预演详情页 20 秒加载超时；该问题第二次已成功，属于独立的页面加载稳定性问题。
- 不自动重试点击沟通。
- 不扫描或接管用户原有的 BOSS 历史会话。
- 不执行真实投递作为自动测试。

## 完成标准

在同标签或新标签跳转中，消息通道因详情页卸载而关闭时，JobCopilot 能通过实际聊天 URL 和岗位 ID 确认正确目标页，然后才注入三段发送脚本；真实未跳转、岗位 ID 不一致或脚本失败时仍立即停止批次。
