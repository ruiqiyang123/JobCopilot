# JobCopilot 预演与投递前置链路韧性设计

日期：2026-07-15

## 背景

在连续预演和正式投递较多岗位时，真实运行出现三类互相独立、但目前都会终止整批的问题：

1. 岗位详情页后台加载超过固定 20 秒后，被统一报告为“页面加载超时”。BOSS 页面即使已经渲染出完整 JD，只要标签页仍未进入 Chrome 的 `complete` 状态，也会被误判为失败。
2. LongCat 在生成个性化开场时偶尔超过默认 30 秒，返回“模型请求超时”。当前不会自动重试，并会让后续已批准岗位全部变为未执行。
3. 点击沟通后，BOSS 偶尔进入不带 `jobId` 查询参数的聊天路由。当前只能从 URL 核验岗位身份，因此即使聊天页已经展示了对应职位，也会为了防止发错 HR 而停止。

现有流程还把任意一个岗位的前置失败当成整个批次失败。批次越大，至少遇到一次瞬时故障的概率越高，用户需要反复从第一个岗位重新运行。

## 目标

- 以“完整 JD 已可读取”作为详情页就绪条件，不再依赖整页 `complete`。
- 对详情页瞬时加载失败和 AI 开场生成超时各进行一次有上限的自动重试。
- URL 缺少岗位 ID 时，允许使用聊天页中可验证的稳定岗位 ID 作为辅助身份依据。
- 任何身份依据都必须精确匹配原岗位 ID；身份不明确时绝不发送。
- 尚未进入消息发送阶段的单岗位失败只影响当前岗位，批次继续处理后续岗位。
- 支持仅重试失败岗位，避免重复预演或重复联系已成功岗位。
- 日志准确区分页面、登录、岗位状态、模型和聊天身份故障。

## 已选方案

采用用户批准的“JD 就绪检测 + 有限重试 + 单岗失败继续整批 + 稳定 ID 辅助核验”组合。

### 方案对比

#### 详情页加载

- **只把 20 秒改成 60 秒**：实现简单，但 JD 已出现时仍会白等，后台资源长期不结束时仍会误判，不采用。
- **把详情标签改为前台激活**：可能减少 Chrome 对后台标签的降速，但会不断抢占用户页面，不采用。
- **轮询 JD 就绪并有限重开（采用）**：以业务内容为准，第一次失败后只重新打开一次，不无限刷新。

#### 聊天身份

- **缺少 URL 岗位 ID 时直接发送**：有发给错误 HR 的风险，不采用。
- **只延长 URL 等待时间**：可以覆盖部分慢跳转，但无法兼容始终不带查询参数的新路由，不作为唯一方案。
- **URL ID 或页面稳定 ID 二选一精确匹配（采用）**：兼容路由变化，同时保持岗位身份安全边界。

#### 批次失败

- **任意失败立即停止整批**：安全但对尚未发送的瞬时错误过度保守。
- **所有失败都继续下一岗位**：发送过程中失败后继续可能造成部分发送和重复投递，不采用。
- **按是否开始发送分界（采用）**：发送前失败跳过当前岗位；发送开始后失败立即停止整批。

## 核心状态与边界

每个岗位的预演或正式投递前置流程记录以下阶段：

```text
queued
opening_detail
waiting_for_jd
generating_opening
verifying_job
opening_chat
verifying_chat_identity
send_bundle_started
succeeded | failed | not_run
```

`send_bundle_started` 是批次容错边界：

- 在该阶段之前失败，确认没有发送任何文本或图片，当前岗位标记失败并继续下一个岗位。
- 进入该阶段之后发生任何错误，保留现有“立即停止整个批次”规则。
- 已成功岗位不会因为重试失败岗位而再次执行。
- 未确认预演的岗位始终不能进入正式投递。

## 组件设计

### 1. 详情页业务就绪检测

新增详情页加载协调器，替代 `createDetailTab` 中单纯等待 `tabs.status === complete` 的逻辑。

每次尝试执行：

1. 打开规范化后的岗位详情 URL。
2. 在标签页存在且 URL 仍属于 BOSS 的前提下，周期性尝试注入详情读取脚本。
3. 内容脚本检查页面状态并返回以下结果之一：
   - `ready`：已获得非空完整 JD 和当前岗位信息；
   - `loading`：页面存在，但完整 JD 尚未出现；
   - `login_required`：出现登录页、登录弹窗或明确登录提示；
   - `risk_control`：出现安全验证、访问异常或人机验证；
   - `job_unavailable`：岗位下架、停止招聘或页面不存在；
   - `invalid_page`：跳转到非预期页面。
4. 一旦返回 `ready`，立即进入身份和资格校验，不再等待图片、统计接口或整页 `complete`。
5. 明确的登录、风控、岗位下架和无效页面错误不自动刷新，并提供对应中文提示。

时间策略：

- 第一次尝试最长 20 秒。
- 只有持续 `loading`、标签页网络错误或无法注入内容脚本时，关闭临时标签，等待 1–2 秒后重新打开一次。
- 第二次尝试最长 35 秒。
- 两次均失败后把当前岗位标记为详情加载失败，并继续批次。
- 不对同一个岗位进行第三次自动尝试，不无限刷新。

连续岗位之间保留短暂节奏间隔，避免瞬间创建大量标签页。任何临时详情标签在成功、失败或取消后都必须关闭。

### 2. AI 个性化开场有限重试

本次只处理预演中的个性化开场生成，不改变快速筛选和详细评分的产品逻辑。

- 第一次调用沿用当前请求配置和 30 秒截止时间。
- 只有 `模型请求超时`、网络中断或模型服务 5xx 可自动重试。
- API Key 错误、权限错误、额度不足、响应格式错误和用户取消不重试。
- 自动重试前等待约 1 秒，第二次最长等待 45 秒。
- 第二次成功后只保存最终一次生成的开场，仍需用户确认预演。
- 第二次仍失败时当前岗位预演失败，后续岗位继续。
- 日志显示 `AI 开场生成超时，正在重试 1/1`，避免用户误以为界面卡死。

模型重试不会进入正式发送，也不会自动批准或确认预演。

### 3. 聊天页稳定岗位身份核验

聊天身份可以来自两种强证据：

1. 聊天 URL 的 `jobId` 与原岗位稳定 ID 完全一致；或
2. 当前聊天页职位卡片、职位详情链接或明确的 `data-job-id` 中提取出的稳定岗位 ID，与原岗位 ID 完全一致。

核验规则：

- URL 包含岗位 ID 时仍以 URL 为首要依据。
- URL 与页面都包含 ID 且二者不一致时立即失败。
- URL 缺少 ID 时，在现有总建联时间内等待页面职位卡片出现，再读取稳定 ID。
- 只根据岗位名称、公司名称或 HR 姓名相同不能通过核验。
- 页面中出现多个不同岗位 ID、无法唯一定位当前职位卡片或 ID 不一致时立即失败。
- 身份通过前不得注入发送脚本、填写输入框、上传图片或点击发送。

首次身份核验失败且尚未发送时，继续沿用现有一次性重新打开机制。第二次仍无法确认身份时，当前岗位标记失败并继续下一个岗位。

### 4. 批次继续与失败岗位重试

预演批次：

- 详情页、AI 开场或资格复核失败时，保存当前岗位的具体错误并继续后续岗位。
- 成功岗位保留已生成预演；失败岗位不产生可确认预演。
- 批次汇总显示成功、失败和未执行数量。

正式投递批次：

- 详情加载、预演有效性复核、岗位资格复核、建联或聊天身份核验失败时，确认 `send_bundle_started` 尚未发生，记录失败并继续下一岗位。
- 一旦调用 `SEND_BUNDLE`，任何文本或图片发送错误都停止整批。
- 用户主动停止、登录失效和 BOSS 风控验证属于全局阻断，立即停止整批，不继续下一岗位。
- 成功岗位继续按现有规则移出当前批次并进入岗位进度。

新增“重试失败岗位”动作：

- 预演区域只选择最近一次预演失败且仍为已批准的岗位。
- 正式投递区域只选择最近一次发送前失败、预演仍有效且尚未成功投递的岗位。
- 已成功、已经开始过发送但结果不明确、已取消批准和已处理岗位不会被自动重试。
- 点击后仍展示岗位数量并要求使用现有正式投递确认流程。

### 5. 错误分类与日志

替换笼统的“请检查网络或重新登录 BOSS”，至少区分：

- `岗位详情加载较慢，自动重试 1/1`
- `岗位详情在两次尝试后仍未就绪`
- `BOSS 登录已失效，请重新登录后重试`
- `BOSS 出现安全验证，已停止整个批次`
- `岗位已下架或停止招聘`
- `AI 开场生成超时，自动重试 1/1`
- `AI 开场两次请求均超时`
- `聊天 URL 缺少岗位 ID，正在读取页面职位卡片`
- `聊天页岗位身份已通过页面职位卡片确认`
- `聊天页无法确认对应岗位，当前岗位未发送`

日志必须标明当前岗位序号、岗位名称和当前重试次数。不得记录 API Key、完整简历图片或完整模型请求正文。

## 数据与持久化

最近一次批次为每个岗位保存：

```text
jobId
stage
attempts.detail
attempts.aiOpening
attempts.chatIdentity
errorCode
errorMessage
sendStarted
updatedAt
```

- `sendStarted` 只能在实际调用 `SEND_BUNDLE` 前一刻置为 `true`。
- Service Worker 重启后，`sendStarted === true` 且没有明确成功结果的岗位不得自动重试。
- 失败岗位重试开始时创建新的运行记录，不覆盖上一轮错误审计信息。
- 不把完整 JD、API Key 或图片 Base64 复制进批次错误记录。

## 测试

### 详情页加载

- 标签页未 `complete`，但内容脚本已经读取到完整 JD 时立即成功。
- 第一次持续 loading，第二次读取成功时只重开一次。
- 两次持续 loading 后当前岗位失败，后续岗位继续。
- 登录失效、风控验证和岗位下架不执行自动刷新。
- 所有临时详情标签在结束后被关闭。

### 模型重试

- 第一次超时、第二次成功时生成一个可审核预演。
- 两次超时后当前岗位失败，后续岗位继续。
- 401、403、429、格式错误和取消不会自动重试。
- 重试日志不包含密钥或完整输入。

### 聊天身份

- URL ID 正确时沿用现有成功路径。
- URL 无 ID、页面职位卡片 ID 正确时通过。
- URL 与页面 ID 冲突时停止当前岗位且不调用发送。
- 页面只有相同岗位名但没有稳定 ID 时不通过。
- 首次缺少身份、重新打开后成功时只重试一次。
- 两次均无法确认时后续岗位继续，且当前岗位没有任何已发送消息。

### 批次语义

- 发送前单岗失败不会把剩余岗位标记为未执行。
- 登录失效、风控或用户停止会终止整批。
- `SEND_BUNDLE` 开始后的失败仍立即终止整批。
- “重试失败岗位”不包含已成功或发送结果不明确的岗位。
- Service Worker 重启后不会自动重试 `sendStarted` 岗位。

### 回归验证

- 全部既有自动测试通过。
- 新增加载协调器、模型重试、页面身份提取和批次继续测试。
- JavaScript 语法、Manifest JSON、差异格式和 API Key 扫描通过。
- 使用 Chrome API、DOM 和模型请求 Mock 验收；不访问 BOSS、不执行真实投递。

## 非目标

- 不取消预演确认或正式投递批次确认。
- 不仅凭岗位名、公司名或 HR 名称确认聊天身份。
- 不无限刷新页面或无限重试模型。
- 不在发送结果不明确时自动重试当前岗位。
- 不放宽登录失效和风控页面的全局停止规则。
- 不修改岗位筛选、AI 详细评分或招呼语内容规范。
- 不在本规格中修改简历图片上传与发送确认；图片可靠性作为下一项独立修复。
- 不使用真实账号投递作为自动测试。

## 完成标准

连续预演或投递多个岗位时，插件以完整 JD 是否可读取判断详情页就绪；页面和 AI 的瞬时超时各最多自动重试一次。发送前的单岗位失败不会中断剩余岗位，登录、风控和发送开始后的失败仍安全停止。聊天 URL 缺少岗位 ID 时，插件只有在页面职位卡片提供同一稳定岗位 ID 后才能继续，无法确认时不发送。用户可以只重试失败岗位，且不会重复执行已经成功或发送结果不明确的岗位。
