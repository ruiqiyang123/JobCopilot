# JobCopilot 人工批准队列与三段式投递设计

日期：2026-07-14
状态：已完成对话评审，等待书面规格确认

## 1. 背景

当前版本已经支持岗位收集、经验与公司规模硬筛选、AI 筛选、模拟运行、正式投递和岗位进度管理。真实使用暴露出以下问题：

1. 预演会重新加载 BOSS 搜索页，并且只在当前已经渲染的岗位卡片中定位目标岗位。排序变化、懒加载或页面刷新后容易出现“未找到岗位卡片”。
2. 公司名称和公司规模解析不稳定，审核区出现较多“公司未知”“规模未知”。
3. 审核区只能勾选岗位，缺少方便的完整岗位入口以及明确的“已批准/不投递”决策。
4. 当前投递消息为“简历图片 + AI 招呼语”，与用户实际使用的三段式沟通方式不一致。
5. 模拟运行与正式投递依赖模式下拉框和重复勾选，用户不容易判断哪些岗位已经人工审核、哪些内容将被实际发送。

本次设计将流程调整为：AI 完成初筛后，由用户查看并明确批准岗位，再对已批准岗位执行可预览、可冻结、可追溯的批量自动投递。

## 2. 目标

### 2.1 核心目标

- 稳定地通过岗位详情链接和岗位 ID 读取完整 JD，不再依赖搜索页当前可见卡片。
- 在硬筛选后使用完整 JD 和用户简历进行 AI 筛选。
- 为筛选结果提供明确的人工审核队列。
- 只有人工批准并完成预演确认的岗位可以正式投递。
- 支持可保存、可选择的招呼方案。
- 正式投递按固定顺序发送：AI 个性化开场、固定补充消息、简历图片。
- 任一步失败立即停止批次，不自动重试。

### 2.2 非目标

- 不扫描或导入用户既有的 BOSS 历史沟通记录。
- 不读取 BOSS 内置常用语列表。
- 不自动批准 AI 推荐岗位。
- 不绕过登录、验证码、风控或平台安全提示。
- 第一版不允许自由拖拽消息顺序。
- 第一版不跨浏览器同步配置和审核记录。

## 3. 用户流程

完整流程如下：

```text
配置模型、简历、筛选条件和招呼方案
  → 收集岗位
  → 岗位卡片硬筛选
  → 稳定链接读取完整 JD 并补全字段
  → AI 结合简历和完整 JD 筛选
  → 用户查看岗位详情
  → 用户明确批准或拒绝
  → 为已批准岗位生成三段式预演
  → 用户编辑并确认 AI 开场
  → 批次确认
  → 逐个自动投递
  → 更新岗位进度和批次结果
```

### 3.1 配置

用户配置：

- AI 服务商、API Key、Base URL 和模型。
- 简历文字。
- 搜索关键词、城市和收集数量。
- 工作经验和公司规模筛选。
- 一套或多套招呼方案。

API Key 继续只保存在 `chrome.storage.local`，不写入源码、测试、快照或日志。

### 3.2 收集与筛选

1. 插件按关键词和城市打开 BOSS 搜索页并收集岗位。
2. 收集时保存岗位 ID、规范化详情链接和卡片快照。
3. 先执行经验与公司规模硬筛选。
4. 卡片信息缺失时，先读取详情页并重新解析，不立即进入人工待确认。
5. 详情页仍无法补全启用的硬筛选字段时，岗位进入“待补充”。
6. 通过硬筛选的岗位使用完整 JD 和简历文字进行 AI 筛选。
7. AI 只产生推荐结论与理由，不产生人工批准。

### 3.3 人工审核

岗位审核区按以下标签展示：

- 推荐岗位
- 待补充
- 已批准
- 不投递
- 已排除

每张岗位卡展示：

- 岗位名称、公司、薪资。
- 工作经验、公司规模。
- JD 摘要。
- AI 推荐结论和理由。
- “查看完整岗位”“批准投递”“不投递”操作。

“查看完整岗位”与“批准投递”是两个独立动作。点击查看不会自动批准。用户必须显式批准岗位，才会形成可投递队列。

### 3.4 预演

预演只处理已批准岗位，并执行：

1. 重新读取并验证岗位身份和可用性。
2. 根据完整 JD、简历和招呼方案生成 AI 个性化开场。
3. 组合固定补充消息和简历图片。
4. 展示最终三段内容。
5. 允许用户针对单个岗位编辑 AI 开场。
6. 用户点击“确认预演”后冻结发送内容。

预演不得点击“立即沟通”，不得进入发送流程，也不得向 HR 发送任何内容。

如果招呼方案、简历文字、简历图片、岗位身份或完整 JD 在确认后发生变化，原预演标记为过期，必须重新预演。

### 3.5 正式投递

正式投递入口只在存在“已批准且预演已确认”的岗位时启用。按钮直接显示可执行数量，例如“正式投递 3 个已批准岗位”，不再依赖模拟/正式模式下拉框。

用户在批次确认框中查看岗位名单和选定招呼方案。确认后，插件对每个岗位依次执行：

1. 重新读取岗位详情。
2. 校验岗位身份、可用性、筛选资格和去重状态。
3. 点击“立即沟通”并进入对应聊天。
4. 发送冻结后的 AI 个性化开场。
5. 确认第一段发送成功。
6. 发送固定网站与 Agent 补充消息。
7. 确认第二段发送成功。
8. 上传并发送简历图片。
9. 确认图片发送成功。
10. 标记岗位为“已沟通”，处理下一个岗位。

当前 BOSS 账号的系统默认招呼语已由用户关闭，因此插件可以完整控制消息顺序，不需要处理重复默认消息。

## 4. 稳定岗位定位

### 4.1 岗位身份

收集记录新增或稳定保存：

```js
{
  id: string,
  detailUrl: string,
  name: string,
  company: string,
  salary: string,
  experience: string,
  companySize: string,
  rawFacts: string[],
  collectedAt: number
}
```

`detailUrl` 必须规范化，移除不影响岗位身份的查询参数。优先使用 URL 中的稳定岗位 ID。只有缺少稳定 ID 时，才使用岗位名称、公司和链接的组合生成后备身份。

### 4.2 详情解析顺序

1. 优先在后台临时标签页直接打开保存的 `detailUrl`。
2. 从岗位详情页解析岗位身份、完整 JD、公司、经验和规模。
3. 读取完成后关闭临时标签页，不改变用户当前搜索页。
4. 详情链接缺失时，回退到搜索页，增量滚动并按稳定 ID 查找。
5. 稳定 ID 不存在时，才允许使用严格的“岗位名称 + 公司”匹配。

不得仅凭相似岗位名称执行正式投递。身份无法证明一致时，岗位进入错误状态并停止批次。

### 4.3 字段解析

解析器按明确来源分层：

1. 岗位卡片的专用名称、公司、标签和规模节点。
2. 岗位详情页的专用标题、公司信息和职位描述节点。
3. 限定范围内的可见文字解析。

避免使用过宽的 `[class*="company"] a` 作为首选公司选择器。解析结果保留原始文本和来源，便于错误提示和测试。

## 5. 招呼方案

### 5.1 数据结构

```js
{
  id: string,
  name: string,
  aiOpeningEnabled: boolean,
  aiInstruction: string,
  fixedMessageEnabled: boolean,
  fixedMessage: string,
  resumeImageEnabled: boolean,
  resumeImage: string,
  createdAt: number,
  updatedAt: number
}
```

第一版允许保存、编辑、删除和选择多套招呼方案。至少保留一套方案时才允许正式投递。删除当前正在被未完成预演使用的方案时，相关预演自动过期。

### 5.2 AI 个性化开场

默认生成规则：

- 依据完整 JD、简历文字和真实个人优势。
- 选择岗位最重要且用户真实具备的 3 至 5 个匹配点。
- 不虚构项目、技能、年限或成果。
- 使用适合 BOSS 聊天的自然中文。
- 避免机械堆砌关键词和空泛自我评价。
- 结尾自然表达交流意愿。

用户可以编辑方案级生成指令，也可以在预演中编辑单个岗位的最终 AI 开场。

### 5.3 固定消息与图片

固定补充消息作为独立聊天消息发送，用于个人网站、个人 Agent 或其他补充说明。简历图片作为第三段发送。三个部分可以单独关闭，但启用部分的相对顺序固定为：

```text
AI 个性化开场 → 固定补充消息 → 简历图片
```

## 6. 预演冻结与过期

每个岗位的预演记录包含：

```js
{
  jobId: string,
  greetingPlanId: string,
  aiOpening: string,
  fixedMessage: string,
  resumeImageFingerprint: string,
  jdFingerprint: string,
  configFingerprint: string,
  status: 'draft' | 'confirmed' | 'expired' | 'failed',
  confirmedAt: number,
  error: string
}
```

正式投递只读取 `confirmed` 状态下的冻结内容，不重新调用模型。以下任一变化会将预演设为 `expired`：

- 招呼方案内容变化。
- 简历文字变化。
- 简历图片变化。
- 岗位 ID、公司或岗位名称变化。
- 完整 JD 指纹变化。
- 相关筛选配置发生影响资格的变化。

## 7. 状态模型

审核状态和求职进度分开保存。

### 7.1 审核状态

```text
pending_review  待审核
needs_info      待补充
approved        已批准
rejected        不投递
filtered_out    已排除
```

### 7.2 求职进度

```text
collected   已收集
contacted   已沟通
replied     已回复
interview   面试
rejected    淘汰
offer       Offer
```

审核状态 `rejected` 与求职进度 `rejected` 存储在不同字段中，避免“决定不投递”和“投递后被淘汰”混淆。

### 7.3 批次状态

批次保存：

- 请求岗位列表。
- 实际执行岗位列表。
- 当前岗位和当前发送步骤。
- 已成功、失败和未执行岗位。
- 开始、停止和完成时间。
- 使用的招呼方案 ID。

浏览器刷新或 Service Worker 重启后可以恢复审核、预演和历史结果，但不得自动恢复正在发送的批次。中断批次保持停止状态，必须由用户重新检查后创建新批次。

## 8. 界面设计

侧边栏按以下区域组织：

1. 配置。
2. 招呼方案。
3. 收集与筛选。
4. 岗位审核。
5. 投递预演。
6. 批量投递。
7. 岗位进度。
8. 可折叠日志。

审核区使用带数量的标签页。已批准队列提供集中预演入口。正式投递按钮不与模拟运行共用同一个按钮，且在条件不满足时显示具体禁用原因。

日志默认折叠。发生错误时自动展开并定位到失败岗位与失败步骤。

## 9. 安全门与错误处理

### 9.1 正式投递安全门

每个岗位必须同时满足：

- 审核状态为 `approved`。
- 预演状态为 `confirmed`。
- 预演没有过期。
- 岗位身份验证通过。
- 岗位仍然可用。
- 硬筛选仍然通过，或者存在明确人工覆盖。
- 尚未成功投递。

### 9.2 失败策略

以下任一步失败，立即停止整个批次：

- 详情读取。
- 身份校验。
- 岗位可用性校验。
- 建立沟通。
- AI 开场发送确认。
- 固定补充消息发送确认。
- 简历图片上传或发送确认。

失败岗位记录错误，剩余岗位记录为未执行。插件不自动重试，也不跳过失败岗位继续发送。

错误信息必须描述阶段和原因，例如：

- `详情读取失败：岗位已下线`
- `身份校验失败：公司名称已变化`
- `发送失败：固定补充消息未确认送达`
- `发送失败：简历图片上传超时`

## 10. 数据迁移

- 保留现有模型配置、筛选配置、简历、岗位进度和已投递去重记录。
- 现有 `sw_previews` 不具备新冻结字段，统一迁移为过期状态，不允许直接正式投递。
- 现有已收集岗位迁移为 `pending_review`，已被硬筛选排除的岗位迁移为 `filtered_out`。
- 现有已沟通、已回复、面试、淘汰和 Offer 状态保持不变。
- 迁移失败不得清空原存储；记录可理解的错误并保持正式投递禁用。

## 11. 测试设计

### 11.1 单元测试

覆盖：

- 岗位 URL 规范化和稳定 ID 提取。
- 卡片与详情页公司、经验和规模解析。
- 审核状态转换。
- 招呼方案校验、保存、切换和删除。
- 预演内容冻结、编辑和过期。
- 重复岗位去重。
- 三段消息顺序。
- 正式投递安全门。

DOM 解析测试使用去标识化的 BOSS 卡片和详情页 HTML 片段，不依赖实时网站。

### 11.2 模拟集成测试

使用 Mock 页面覆盖完整流程：

```text
收集 → 详情读取 → 硬筛选 → AI 筛选 → 人工批准
→ 预演 → 确认 → 建立沟通 → 三段发送 → 记录成功
```

分别模拟第一段、第二段和图片发送失败，验证批次立即停止且剩余岗位标记为未执行。

### 11.3 真实页面模拟运行

在用户已登录的 BOSS 页面验证：

- 稳定链接能够读取完整详情。
- 当前页面不会因预演被破坏。
- 公司、经验和规模能够在页面有对应信息时被读取。
- 审核状态和预演内容在刷新后保留。
- 模拟运行不点击“立即沟通”。

### 11.4 单岗位真实验收

只有模拟运行全部通过，并获得用户对具体岗位的单独授权后，才执行一次单岗位真实投递。验证三段内容、顺序、送达确认和进度记录。

## 12. 验收标准

- 搜索页刷新或排序变化不再导致常规的“未找到岗位卡片”。
- 当 BOSS 页面存在公司和规模信息时，插件能正确读取并展示。
- AI 筛选使用完整 JD 和用户简历。
- 用户可以打开完整岗位并显式批准或拒绝。
- 未批准岗位不能预演或正式投递。
- 正式发送内容与确认预演完全一致。
- 正式发送顺序为 AI 开场、固定补充消息、简历图片。
- 任一步失败立即停止，不自动重试。
- 刷新浏览器后审核、预演和岗位进度仍然保留。
- 旧版配置和岗位进度完成迁移。
- 仓库和日志不包含 API Key。
