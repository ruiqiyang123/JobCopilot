# JobCopilot 发送前建联自动重试设计

日期：2026-07-15

## 背景

真实投递中，同一批岗位第一次可能在 BOSS 聊天壳停留超过 5 秒并报“聊天页缺少岗位 ID”，用户立即重新运行后又能正常取得岗位 ID 并完成投递。这说明当前 5 秒身份宽限偏短，而且失败发生在三段消息发送之前，用户只能手动重新运行整批。

## 目标

- 将相关聊天页缺少 `jobId` 的宽限时间从 5 秒增加到 12 秒。
- 宽限后仍缺少岗位 ID 时，只对当前岗位的建联阶段自动重试一次。
- 自动重试前重新打开岗位详情并重新执行身份、筛选、预演和去重校验。
- 任何消息开始发送后都不自动重试，避免重复招呼或重复图片。
- 第二次建联仍失败时立即停止批次，保持现有失败即停规则。

## 方案比较

### 方案 A：只增加等待时间

改动最小，但聊天壳真正卡住时仍需要用户手动重新运行。

### 方案 B：12 秒宽限 + 发送前重试一次（采用）

覆盖 BOSS 偶发慢加载，也能自动恢复一次卡死的聊天壳。重试严格限定在 `SEND_BUNDLE` 之前，安全边界清楚。

### 方案 C：岗位或批次发送失败后自动重试

可能在第一段或第二段已经送达时重复发送，不采用。

## 重试条件

只有同时满足以下条件才允许自动重试：

- 当前错误代码为 `missing_job_id`。
- 当前岗位尚未调用 `SEND_BUNDLE`。
- 当前岗位本批次尚未自动重试。
- 批次未被用户暂停或停止。

`job_mismatch`、按钮缺失、订阅弹窗无法关闭、页面脚本失败、模型失败、资格变化和任一消息发送失败都不得自动重试。

## 数据流

1. 打开当前岗位详情页，读取并校验岗位身份、可用状态、经验、公司规模、重复投递和预演指纹。
2. 启动聊天页观察器并点击沟通。
3. 相关聊天页短暂缺少 `jobId` 时等待最多 12 秒。
4. 若取得正确岗位 ID，进入聊天页并发送三段内容。
5. 若 12 秒后仍缺少 ID，确认尚未调用 `SEND_BUNDLE`，记录“聊天身份加载失败，正在自动重试 1/1”。
6. 保留并最终清理第一次任务创建的临时标签页；重新打开岗位详情。
7. 重新执行第 1 至 4 步，不复用第一次的页面身份结果。
8. 第二次仍失败时停止整个批次；成功时只发送一次三段内容。

## 状态与日志

- `currentStep` 在重新打开详情时回到 `detail`，进入第二次建联时再变为 `contact`。
- `executedIds` 每个岗位只记录一次，不因建联重试重复计数。
- 新增日志：`聊天身份加载失败，正在自动重试 1/1`。
- 成功日志和失败汇总保持现有格式。
- 不记录聊天 URL、`securityId`、消息正文或 API Key。

## 标签页与清理

- 第一次为任务新建的详情/聊天标签继续加入 `temporaryTabIds`。
- 重试重新打开的临时详情/聊天标签同样加入集合。
- 岗位成功、失败或批次中止后统一按标签 ID 去重关闭。
- 用户原本打开的搜索、详情和聊天标签不得自动关闭。

## 测试

- 无岗位 ID 在 12 秒内补齐时不重试并正常发送。
- 第一次返回 `missing_job_id`、第二次身份匹配时只发送一次。
- 第二次仍缺少岗位 ID 时停止批次。
- `job_mismatch` 和其他建联错误不重试。
- `SEND_BUNDLE` 任一步失败不重试。
- 重试前重新执行岗位详情与预演校验。
- 同一岗位在 `executedIds` 中只出现一次。
- 两次任务创建的临时标签页都被清理，用户已有标签不关闭。
- 全部现有测试、JavaScript 语法、Manifest、差异和密钥检查通过。
- 不使用真实投递作为自动化测试。

## 非目标

- 不无限延长聊天身份等待。
- 不自动重试岗位身份不一致或风控/登录页面。
- 不自动重试任何文字或图片发送。
- 不继续执行当前批次的后续岗位来绕过失败。
- 不降低岗位 ID 精确匹配和发送前资格校验。

## 完成标准

BOSS 聊天身份慢加载时插件最多等待 12 秒；仍缺少岗位 ID 时，在尚未发送任何消息的前提下自动重新打开并完整校验当前岗位一次。成功只发送一组三段消息，第二次失败则安全停止批次。
