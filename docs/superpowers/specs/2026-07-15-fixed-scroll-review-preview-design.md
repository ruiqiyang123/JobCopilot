# 岗位审核与投递预演固定滚动窗口设计

## 背景

一次筛选可能产生五六十个岗位。当前“④ 岗位审核”和“⑤ 投递预演”的列表会随内容持续增长，用户需要在整个侧边栏中反复长距离滚动，才能在岗位内容、分类、批量操作和下一步按钮之间往返。

## 目标

- 为岗位审核和投递预演分别提供独立的固定高度窗口。
- 只滚动岗位卡片或预演卡片，关键控制区始终可见。
- 在常见侧边栏高度下同时兼顾列表容量和上下文可见性。
- 保持现有筛选、审核、预演和正式投递业务规则不变。

## 非目标

- 不增加分页、虚拟列表、搜索或新的筛选条件。
- 不压缩或重新设计现有岗位卡片和预演卡片内容。
- 不改变批量确认、批量批准、预演确认或正式投递的数据协议。
- 不改变正式投递前的最终确认弹窗。

## 选定方案

采用“顶部与底部固定，中间列表独立滚动”的方案。

两个窗口都使用响应式高度：`height: clamp(360px, 60vh, 720px)`。高度约占侧边栏可视区域的 60%，在较矮窗口中不低于 360px，在较高窗口中不超过 720px。

### 岗位审核窗口

窗口从上到下分为三个区域：

1. 固定顶部控制区
   - 岗位审核标题和总数。
   - 推荐、待补充、已批准、不投递、已排除分类标签。
   - 当前分类对应的全选和一键操作。
2. 可滚动内容区
   - 仅包含当前分类下的岗位卡片。
   - 空分类时在滚动区内显示空状态。
3. 固定底部操作区
   - 已批准岗位数量。
   - “预演已批准岗位”按钮。

### 投递预演窗口

窗口从上到下分为三个区域：

1. 固定顶部控制区
   - 投递预演标题和已确认数量。
   - 安全提示、预演错误与重试入口。
   - 全选待确认预演和“一键确认预演”。
2. 可滚动内容区
   - 仅包含各岗位的预演卡片及可编辑开场文本。
   - 没有已批准岗位时在滚动区内显示空状态。
3. 固定底部操作区
   - 正式投递按钮。
   - 无法投递时的原因提示。
   - 批次完成后的后续操作区域。

## 页面结构

两个卡片使用同一种布局边界：

- 固定窗口
  - 固定顶部控制区
  - 可滚动列表（`flex: 1; min-height: 0; overflow-y: auto`）
  - 固定底部操作区

固定窗口使用纵向 Flex 布局。滚动列表必须同时设置 `min-height: 0` 和 `overflow-y: auto`，确保列表在窗口内部收缩并产生滚动，而不是继续撑高整个侧边栏。

滚动区增加：

- `overscroll-behavior: contain`，避免滚到边界后继续带动整个侧边栏。
- `scrollbar-gutter: stable`，避免滚动条出现或消失时卡片宽度跳动。
- 清晰但克制的边界或阴影，提示列表是可独立滚动的。

## 交互规则

- 鼠标滚轮、触控板和键盘滚动焦点位于列表时，只滚动当前列表。
- 切换岗位审核分类后，新分类从列表顶部开始显示。
- 单岗位勾选、人工确认、批准、拒绝或异步状态更新后，保留当前分类的滚动位置。
- 批量操作导致当前分类岗位数量减少时，将滚动位置约束在新的有效范围内，不能出现大段空白。
- 预演文本编辑、单岗位重新生成或确认后，保留预演列表滚动位置。
- 列表为空时，空状态位于可滚动内容区，顶部和底部控制区仍保持稳定。
- 页面跳转到岗位审核或投递预演卡片时，只定位外层卡片，不主动改变内部列表位置，分类切换除外。

## 状态管理

不新增持久化数据。滚动位置只属于当前侧边栏会话的界面状态：

- 岗位审核记录当前可见分类的滚动位置，用于同一分类重渲染后恢复。
- 投递预演记录一个滚动位置，用于重渲染后恢复。
- 切换岗位审核分类时明确将新分类位置重置为顶部。
- 重新打开侧边栏时两个列表都从顶部开始。

渲染函数在替换列表内容前读取 `scrollTop`，渲染后在下一帧恢复并约束到有效范围。分类点击处理明确请求下一次渲染重置到顶部，不恢复上一个分类的位置。

## 无障碍与可用性

- 滚动区可通过键盘聚焦，并提供描述其用途的可访问名称。
- 保持现有按钮的禁用、加载和错误状态。
- 固定底部区域使用不透明或接近不透明的背景，避免下方列表文字穿透。
- 不能用悬浮层遮挡岗位卡片或预演文本。

## 错误处理

此功能不引入新的网络或业务错误。现有错误仍显示在原有位置：

- 审核操作失败后保持当前滚动位置，并恢复按钮可用状态。
- 预演生成、重新生成或批量确认失败时，错误区域固定在预演窗口顶部，列表位置不跳动。
- 正式投递不可用原因固定在底部按钮附近。

## 验证

### 自动化测试

- 侧边栏静态集成测试确认两个列表都位于固定窗口结构中。
- CSS 契约测试确认窗口具有响应式高度，列表具有 `min-height: 0`、`overflow-y: auto` 和 `overscroll-behavior: contain`。
- 交互测试确认每次切换分类后都从顶部开始。
- 交互测试确认列表重渲染后恢复并约束滚动位置。
- 现有审核、批量确认、预演确认和正式投递测试继续通过。

### 手工验证

- 使用至少 60 个岗位检查审核列表和预演列表。
- 在窄侧边栏、较矮窗口和较高窗口中检查自适应高度。
- 用鼠标滚轮和触控板滚动到列表顶部与底部，确认外层侧边栏不被继续带动。
- 在列表中部执行单岗位和批量操作，确认页面与列表不会跳回顶部。
- 编辑预演文本并确认岗位，确认编辑与滚动互不干扰。

## 验收标准

- 五六十个岗位不会继续把审核或预演卡片无限撑高。
- 用户无需在整个侧边栏中反复长距离滚动即可切换分类、批量操作或进入下一步。
- 两个列表的可用高度会随侧边栏视口变化，并保持在 360–720px 范围内。
- 顶部控制区和底部主操作始终可见。
- 所有现有业务规则和安全确认保持不变。
