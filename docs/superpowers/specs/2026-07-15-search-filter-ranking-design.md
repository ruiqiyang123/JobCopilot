# JobCopilot 批次生命周期、多关键词搜索、可配置筛选与匹配评分设计规格

日期：2026-07-15
状态：用户已批准，实施中（阶段 1–4 已完成）

## 1. 背景

当前插件将单个“岗位关键词”原样写入一次 BOSS 搜索 URL，收集当前结果后，依次读取完整 JD，执行工作经验、公司规模硬筛选和 AI 二值匹配。正式投递成功后只写入防重复记录和岗位进度，没有把岗位从当前审核批次移出；侧边栏关闭后会恢复持久化的审核队列，因此已投岗位继续显示为“已批准”。

这一流程存在四个主要问题：

1. 单一搜索词会遗漏“大模型产品经理”、“智能体产品经理”、“AI 应用产品经理”等语义相关岗位。
2. 筛选维度较少，公司规模全选时实际不产生筛选效果。
3. AI 只返回“匹配/不匹配 + 一句理由”，用户无法看到岗位方向、年限、产品基本功、AI 经历和学历的分项证据。
4. 已成功投递岗位与仍待处理岗位混在当前批次中，开始下一批前需要逐个取消批准。

## 2. 目标

- 通过多关键词扩展收集语义相关的 AI 产品岗位。
- 跨关键词去重，同一岗位只读取一次详情、调用一次 AI。
- 将条件分为硬筛选与软偏好，避免信息缺失导致误排除。
- 将 AI 结果升级为 0–100 分的可解释匹配评分。
- 将当前批次与岗位进度分离，成功岗位自动离开当前审核队列，失败和未执行岗位继续保留。
- 提供批次完成摘要和“开始下一批”，无需逐个取消批准。
- 保留人工审核、预演和投递安全门，不扩大自动投递权限。
- 将筛选条件保存为可复用的命名方案。

## 3. 非目标

- 不通过编辑距离或拼写相似度猜测中文岗位语义。
- 不扫描 BOSS 历史沟通记录。
- 不删除已成功投递的防重复记录和岗位进度；“自动消失”只表示移出当前批次。
- 第一版不提供独立的批次历史看板，成功历史统一进入现有岗位进度。
- 不将筛选评分作为真实投递的唯一决策；正式投递仍必须人工批准和确认预演。
- 不对 BOSS 页面缺失的行业、融资阶段、发布时间等数据进行猜测。

## 4. 设计决策

### 4.1 关键词采用“多查询 + 语义筛选”

插件不在单个 BOSS 搜索页内做字符串模糊匹配，而是按关键词列表依次执行搜索，然后对唯一岗位使用完整 JD 做语义匹配。

默认“AI 产品岗位组”包含：

- AI 产品经理
- 人工智能产品经理
- AI Agent 产品经理
- 智能体产品经理
- 大模型产品经理
- LLM 产品经理
- AI 应用产品经理
- AIGC 产品经理
- AI 平台产品经理
- AI 工具产品经理

用户可添加、删除、停用或恢复关键词。

### 4.2 匹配模式

- **精准**：只搜索用户明确启用的关键词，AI 推荐阈值为 80。
- **平衡（默认）**：搜索用户关键词及内置同义词，AI 推荐阈值为 75。
- **宽松**：在平衡模式基础上加入智能硬件、AI 语音、AI 效率工具等相邻产品方向，AI 推荐阈值为 70。

模式只影响搜索范围和 AI 推荐阈值，不改变硬筛选条件。

### 4.3 收集数量的明确语义

“收集岗位数量”改名为“唯一岗位收集上限”。该数量是所有关键词搜索后、按稳定岗位 ID 去重的总数，不是每个关键词的数量。

为避免前几个关键词占满全部配额，每轮从每个启用关键词最多取 5 个新岗位，循环搜索直到达到总上限或所有搜索结果停滞。

## 5. 当前批次生命周期

### 5.1 状态职责

岗位的人工审核决策与投递结果是两组独立状态：

- `reviewStatus`：`pending_review`、`needs_info`、`approved`、`rejected`、`filtered_out`。
- `deliveryStatus`：`not_started`、`succeeded`、`failed`、`not_run`。

`reviewStatus` 决定岗位是否获得人工批准，`deliveryStatus` 决定岗位是否仍属于当前待处理批次。成功岗位保留原审核决策，但不再参与审核计数、预演和正式投递候选计算。

### 5.2 投递结果处理

每个岗位完成发送后立即更新并持久化状态，同时通知侧边栏刷新：

- **成功**：设置 `deliveryStatus: succeeded` 和 `deliveredAt`，移出当前审核、预演和投递候选列表；写入 `processed` 防重复记录，并在岗位进度中标记“已沟通”。
- **失败**：设置 `deliveryStatus: failed`、`deliveryError` 和失败步骤，保留原 `reviewStatus: approved`，批次按现有安全规则立即停止。
- **未执行**：设置 `deliveryStatus: not_run`，继续保留在已批准列表，用户可在排除失败岗位后继续处理。

失败岗位不自动重试。再次正式投递必须由用户主动操作，且仍需通过岗位身份、预演有效性和防重复校验。

### 5.3 批次完成与下一批

投递结束后显示固定摘要：成功、失败、未执行数量，以及三个操作：

1. `处理剩余岗位`：仅在存在失败或未执行岗位时显示，回到当前批次。
2. `开始下一批`：清理当前批次的岗位、审核和预演数据，回到可收集状态。
3. `查看岗位进度`：定位到现有岗位进度看板。

“开始下一批”保留以下数据：AI 服务商配置、API Key、简历文字、筛选方案、招呼方案、简历图片、`processed` 防重复记录和岗位进度。若当前仍有失败或未执行岗位，必须提示它们会离开当前工作区；若全部成功则直接开始下一批，不再要求逐个取消批准。

### 5.4 恢复与旧数据迁移

侧边栏重新打开或 Service Worker 重启时：

- 从 `processed` 和最近批次的 `succeeded` 补齐旧岗位的 `deliveryStatus: succeeded`。
- 成功岗位不恢复到当前审核列表，但继续存在于岗位进度。
- 失败和未执行岗位继续恢复到当前批次。
- 迁移只补充状态，不删除旧记录，不改变用户配置。

### 5.5 当前批次界面

- “岗位审核”总数和各页签数量只统计仍待处理的岗位。
- 成功岗位从列表即时消失，不需要关闭或刷新侧边栏。
- 全部成功时显示空状态“本批已完成，N 个岗位已移入岗位进度”。
- 部分失败时保留失败与未执行岗位，并在卡片显示最近失败原因。
- 不新增重复的“已投递”审核页签，成功岗位统一从现有岗位进度查看。

## 6. 筛选模型

### 6.1 硬筛选

硬筛选不符合时岗位进入“已排除”，不调用 AI。配置项包含：

1. **工作经验**：沿用现有多选档位，默认“1 年以内”与“1–3 年”。
2. **公司规模**：沿用现有多选档位，新建或重置方案时默认只选 100 人以上。
3. **岗位类型**：全职、实习、兼职，默认只选全职。
4. **学历要求**：不限、大专、本科、硕士、博士；默认允许不限、大专、本科和硕士要求。
5. **薪资范围**：可单独关闭。启用时使用月薪 K 范围与岗位薪资范围求交集；无交集为排除，有交集为通过。薪数仅用于展示，不折算到月薪。
6. **城市与行政区**：城市必选，行政区可多选并可关闭。
7. **发布时间**：不限、当天、3 天内、7 天内、30 天内，默认 7 天内。
8. **必须包含词**：可选，支持多个词，默认为“任意一个命中”。
9. **排除关键词**：可选，用户可分别选择仅匹配岗位标题，或匹配标题与完整 JD。
10. **公司黑名单**：可选，对标准化公司名做精确匹配。

筛选信息缺失时不猜测：

- 该维度已启用且信息缺失：进入“待确认”。
- 用户可人工确认信息缺失项符合，并保存 `manualOverride`。
- 明确不符合硬条件的岗位不允许用人工覆盖绕过；用户需先修改筛选方案。

### 6.2 软偏好

软偏好不排除岗位，只记录命中证据并影响审核页排序：

- 行业偏好：AI、SaaS、智能硬件、汽车、金融科技等可编辑标签。
- 融资阶段或上市状态偏好。
- 公司规模偏好。
- 公司白名单。
- 更新时间偏好。

页面缺少软偏好信息时不扣分、不进入待确认。

## 7. AI 可解释匹配评分

### 7.1 评分维度

AI 必须使用完整 JD 与用户简历事实库，以 JSON 返回：

| 维度 | 满分 | 含义 |
|---|---:|---|
| 岗位方向 | 30 | 是否属于 AI 产品、Agent、大模型应用或相邻产品方向 |
| 工作年限 | 15 | 用户正式经验、实习与项目证据是否支撑 JD 级别 |
| 产品基本功 | 20 | 需求、流程、原型、PRD、协作、验收和数据能力证据 |
| AI 经历 | 20 | LLM、RAG、Agent、Tool Calling、评测、Badcase 和 AI MVP 证据 |
| 学历背景 | 5 | 学历和专业背景是否满足 JD |
| 行业/领域经验 | 10 | 智能硬件、语音、工具、数据或相关行业的可迁移证据 |

综合分必须等于六个分项之和，不允许模型另外生成不可解释的总分。

### 7.2 返回结构

```json
{
  "score": 82,
  "dimensions": {
    "roleDirection": { "score": 27, "max": 30, "evidence": "..." },
    "experience": { "score": 11, "max": 15, "evidence": "..." },
    "productFundamentals": { "score": 17, "max": 20, "evidence": "..." },
    "aiExperience": { "score": 18, "max": 20, "evidence": "..." },
    "education": { "score": 5, "max": 5, "evidence": "..." },
    "domain": { "score": 4, "max": 10, "evidence": "..." }
  },
  "strengths": ["..."],
  "risks": ["..."],
  "reason": "..."
}
```

所有 `evidence`、`strengths`、`risks` 必须引用 JD 或简历事实，不得虚构。后台必须校验分数范围、分项总和和字段类型，再根据当前匹配模式的阈值生成 `matchDecision`；不直接信任模型输出的决策文本。非法 JSON 进入“待确认”，不默认排除。

### 7.3 结果分类

平衡模式默认阈值：

- 75–100：推荐。
- 55–74：待确认。
- 0–54：AI 建议排除。

精准和宽松模式使用 4.2 定义的推荐阈值，“待确认”下限始终为 55。AI 建议排除的岗位可由用户查看完整 JD 后手动加入已批准队列，并记录 AI 决策覆盖；硬筛选明确失败不可直接覆盖。

## 8. 处理流程

1. 读取选中的筛选方案并校验。
2. 按匹配模式生成本次确定的搜索词列表。
3. 按轮次依次打开 BOSS 搜索页并收集岗位卡片。
4. 按稳定岗位 ID 跨关键词去重，保留岗位命中的搜索词列表。
5. 仅对卡片上已明确不符合的条件执行预筛；信息缺失仍读取详情。
6. 对候选岗位读取完整 JD 和公司信息。
7. 用详情信息执行最终硬筛选，生成通过、排除或信息缺失。
8. 仅对硬筛选通过的唯一岗位调用一次 AI 评分。
9. 结合 AI 分数与软偏好命中进行排序，但不修改 AI 原始分数。
10. 审核页按“推荐、待确认、已批准、不投递、已排除”展示，每张卡片展示分项分数、证据、风险和硬筛选理由。

## 9. 界面信息架构

沿用现有侧边栏、白色底、蓝色强调和橙色主操作。不改动模型配置、招呼方案、审核、预演与投递的顶层顺序。

筛选配置分为：

1. **当前方案**：方案选择、新建、重命名、删除、恢复默认。
2. **搜索范围**：关键词标签、自动扩展、精准/平衡/宽松、城市、行政区和唯一岗位上限。
3. **硬筛选**：经验、规模、岗位类型、学历、薪资、时间、必须词和排除词。
4. **偏好排序**：行业、融资阶段、公司偏好和 AI 阈值。
5. **条件摘要**：始终显示高信息密度的一行摘要，例如“深圳｜1 年以内/1–3 年｜100 人以上｜全职｜7 天内｜平衡匹配”。

筛选配置采用用户已确认的“方案优先”组织方式，详细规则见第 16 节。

## 10. 审核卡片

岗位卡片默认展示：

- 综合分与推荐等级。
- 岗位名、公司、薪资、经验、学历、规模和发布时间。
- 最高的 2 个匹配分项与最主要的 1 个风险。
- “查看评分详情”展开六个分项、证据和完整 JD。
- 批准、不投递、手动加入候选等操作。

列表默认按“分类优先级 → 综合分 → 发布时间”排序。

## 12. 错误与边界

- 某个关键词搜索失败：立即停止本次收集，保留已收集岗位和失败关键词，不自动重试。
- 稳定岗位 ID 缺失：保留现有备用身份逻辑，但不将不同公司的同名岗位合并。
- 薪资、学历、发布时间等已启用硬筛选信息缺失：进入待确认。
- AI 超时、非法 JSON 或分数校验失败：进入待确认，显示模型错误，不默认排除。
- 切换筛选方案会使尚未正式投递的旧预演失效；已成功投递记录不受影响。
- API Key 仍只保存于扩展本地存储，不进入筛选方案导出数据。
- 投递成功但岗位进度写入失败时，岗位仍视为已发送并移出当前批次，同时显示进度记录警告，禁止再次发送。
- 侧边栏消息通道关闭不回滚已经成功写入的岗位发送状态。

## 13. 性能与模型消耗

- 先跨关键词去重，再读取详情。
- 卡片明确不符合时可预筛，卡片信息缺失不预筛。
- 最终硬筛选发生在 AI 之前。
- 每个唯一岗位最多调用一次 AI 评分。
- 日志显示搜索词数、原始卡片数、去重数、硬筛数、AI 调用数和最终推荐数。

## 14. 测试与验收

### 单元测试

- 关键词扩展、去重和模式差异。
- 搜索轮询的公平配额与唯一岗位上限。
- 岗位类型、学历、薪资、行政区和发布时间解析。
- 多维硬筛选组合、信息缺失与人工确认。
- AI JSON 的字段、枚举、分数范围和分项总和校验。
- 方案保存、切换、删除、迁移和恢复默认。
- 批次成功、失败、未执行、旧 `processed` 数据迁移和当前岗位过滤。

### 集成测试

- 多关键词逐个构造 BOSS 搜索 URL，数量语义为去重后总上限。
- 同一岗位在多个关键词下出现时只读取详情和调用 AI 一次。
- 硬筛选排除的岗位不调用 AI。
- 推荐、待确认和 AI 建议排除分类正确。
- 审核卡片显示总分、分项证据、风险和排除理由。
- 只有人工批准且预演确认的岗位可正式投递。
- 成功岗位在发送后立即离开当前审核列表，并在重新打开侧边栏后仍不出现。
- 部分失败时只保留失败和未执行岗位，成功岗位不会再次进入投递候选。
- “开始下一批”清理当前批次，但保留配置、招呼方案、岗位进度和防重复记录。

### 静态与数据检查

- 新脚本在 Service Worker 和侧边栏中按依赖顺序加载。
- Manifest 不增加 BOSS 之外的网站权限。
- 仓库、测试快照和日志不包含 API Key。
- 未经用户单独授权不执行真实投递。

## 15. 实施分段建议

1. 当前批次生命周期：成功自动移出、失败/未执行保留、完成摘要、开始下一批、恢复迁移与测试。
2. 关键词组、多搜索、跨搜索去重和方案存储。
3. ✅ 扩展硬筛选字段、解析器、缺失信息处理和 UI。
4. ✅ AI 多维评分、结构校验、排序和审核卡片。
5. 软偏好、公司名单、方案管理和整体回归。

## 16. 已确认的原型方向
