## 📖 项目简介

JobCopilot 是一个运行在 Edge 和 Chrome 侧边栏中的 BOSS 直聘求职辅助扩展。它把原本分散、重复的求职操作串成一条完整流程：

> 多关键词收集岗位 → 读取完整 JD → 硬条件过滤 → AI 匹配判断 → 人工审核批准 → 消息预演确认 → 岗位身份复核 → 逐个发送 → 跟踪沟通进度

它不是无人值守的“海投脚本”。AI 负责整理信息、辅助判断和生成内容，用户始终掌握最终控制权：

- 未经人工批准的岗位不会投递。
- 未确认或已经失效的预演不会发送。
- 无法精确确认岗位身份时不会发送。
- 发送开始后的异常会停止批次，避免在结果不明确时自动重复发送。

## 🔄 工作流程

```mermaid
flowchart LR
    A["配置搜索与招呼方案"] --> B["多关键词收集"]
    B --> C["完整 JD 与硬筛选"]
    C --> D["AI 快速判断"]
    D --> E["人工审核与批准"]
    E --> F["预演、编辑与确认"]
    F --> G["发送前身份复核"]
    G --> H["三段式逐岗发送"]
    H --> I["岗位进度管理"]
```

整个流程分成三个明确阶段：

1. **收集与筛选**：搜索岗位、读取完整 JD，先执行确定性的硬筛选，再调用模型判断匹配度。
2. **审核与预演**：查看岗位详情和筛选理由，明确批准后生成最终消息；内容可以编辑、重新生成并逐岗确认。
3. **发送与跟踪**：发送前再次读取详情并核验稳定岗位 ID，按顺序发送消息，随后更新批次结果和岗位进度。

## ✨ 核心能力

### 岗位收集与筛选

- **多关键词搜索**：关键词支持逗号、顿号、分号或换行分隔，并按稳定岗位 ID 跨搜索去重。
- **三种搜索模式**：精准模式只使用填写词；平衡模式扩展 AI 产品相关词；宽松模式增加相邻产品方向。
- **可复用筛选方案**：整套关键词、城市、搜索模式、数量和硬筛选条件可以命名保存、复制、切换和删除。
- **完整 JD 读取**：保存岗位详情链接，在后台读取完整职位描述，不依赖可能已经刷新的搜索卡片。
- **多维硬筛选**：支持工作经验、公司规模、岗位类型、学历、薪资、城市/行政区、发布时间、必须词、排除词和公司黑名单。
- **可信位置识别**：只读取页面中的独立位置字段；位置缺失时可按本批搜索城市推定，页面明确异地时仍会排除。

### AI 匹配与人工审核

- **多模型支持**：内置 Xiaomi MiMo、DeepSeek、LongCat，也可连接自定义 HTTPS OpenAI 兼容接口。
- **快速 AI 判断**：批量阶段只返回推荐、待确认、排除和一句理由，避免为每个岗位生成冗长报告。
- **按需六维评分**：需要深入判断时，可单独生成岗位方向、工作年限、产品基本功、AI 经历、学历背景和行业经验评分及证据。
- **人工补充与覆盖**：信息不足时进入待补充；硬筛选通过后，用户可以对 AI 的待确认或排除建议作出最终决定。
- **人工批准队列**：推荐、待补充、已批准、不投递和已排除状态分开管理；推荐岗位支持选择后一键批准。

### 招呼方案与消息预演

- **多套招呼方案**：每套方案独立保存 AI 开场规则、固定补充消息和简历图片。
- **三段式消息**：按“AI 个性化开场 → 固定网站/Agent 消息 → 简历图片”的顺序执行，可按需关闭其中某一段。
- **完整预演**：正式发送前展示最终内容，AI 开场可以编辑或针对单个岗位重新生成。
- **预演冻结**：预演会记录岗位、JD、简历、筛选条件和招呼方案的指纹；关键输入变化后必须重新确认。
- **失败岗位重试**：批量预演不会因普通的单岗位错误中断，可只重试预演失败的岗位。

### 安全投递与批次管理

- **发送前复核**：再次读取完整 JD、重新执行筛选，并核对稳定岗位 ID、批准状态和预演状态。
- **聊天页身份核验**：优先读取聊天 URL 的 `jobId`；URL 缺少 ID 时，只接受当前职位卡片中唯一且完全一致的稳定 ID。
- **逐岗位闭环**：一个岗位完成身份确认和消息发送后才会进入下一个岗位。
- **分阶段失败处理**：发送前单岗位失败会记录原因并继续；登录失效、风控或消息开始发送后的失败会停止整批。
- **定向重试**：可以只重试发送前失败的岗位，不会自动重试已经开始发送或结果不明确的岗位。
- **连续批次**：成功岗位立即移出活动队列；失败和未执行岗位保留。完成后可以处理剩余岗位或开始下一批。

### 岗位进度与本地数据

- **岗位进度看板**：管理已收集、已沟通、已回复、面试、淘汰和 Offer 等状态及历史。
- **防重复投递**：已经成功处理的稳定岗位 ID 会保存在本地，后续流程会阻止重复发送。
- **图片单份存储**：简历图片只在每套招呼方案中保存一份，岗位预演只保存图片指纹，避免存储空间随岗位数量增长。
- **实时日志**：展示当前阶段、进度、成功、失败、未执行和图片未确认等结果，支持暂停、继续、停止和重置当前批次。

## 🛡️ 安全与可靠性边界

| 场景 | JobCopilot 的处理方式 |
|------|----------------------|
| 岗位未批准 | 不允许预演或正式发送 |
| 预演未确认、已过期或输入发生变化 | 阻止发送，要求重新预演确认 |
| 详情页仍在加载 | 完整 JD 出现即可继续；持续未就绪时自动重开一次 |
| AI 开场遇到超时或瞬时网络/服务错误 | 有限重试一次；鉴权、额度、配置和格式错误不重试 |
| 聊天 URL 缺少岗位 ID | 读取当前职位卡片；仅在唯一稳定 ID 完全匹配时继续 |
| 发送前单岗位失败 | 跳过当前岗位并继续后续岗位，可稍后只重试这些失败岗位 |
| 登录失效或出现安全验证 | 立即停止整个批次，等待用户处理 |
| 文字消息已经开始发送后失败 | 停止整个批次，不自动重试当前岗位 |
| Service Worker 在发送中重启 | 标记为“结果不明确”，要求先检查 BOSS 会话，不自动重复发送 |
| 文字成功但简历图片未确认 | 记录警告并继续；如果方案只有图片，则仍视为失败 |

这些措施用于减少误投和重复发送，但无法消除网站改版、网络状态、平台限制或自动化操作本身带来的全部风险。

## 🚀 快速开始

### 1. 下载项目

```bash
git clone https://github.com/ruiqiyang123/JobCopilot.git
```

也可以在 GitHub 页面选择 `Code → Download ZIP` 并解压。

### 2. 加载浏览器扩展

1. Edge 打开 `edge://extensions`；Chrome 打开 `chrome://extensions`。
2. 开启右上角的 **开发者模式**。
3. 点击 **加载解压缩的扩展**。
4. 选择仓库中的 **`JobCopilot · AI`** 文件夹，也就是包含 `manifest.json` 的目录。
5. 点击扩展图标，在浏览器侧边栏中打开 JobCopilot。

### 3. 完成基础配置

| 配置项 | 说明 |
|--------|------|
| AI 服务商 | Xiaomi MiMo、DeepSeek、LongCat 或自定义 HTTPS OpenAI 兼容接口 |
| API Key | 用于岗位判断、详细评分和生成个性化开场 |
| API 地址 / 模型 | 内置服务商会自动填充；自定义接口需要手动填写 |
| 简历文字 | 用于完整 JD 匹配、评分和生成开场，不是简历图片 OCR |
| 关键词 / 城市 | 岗位搜索的基础条件 |
| 搜索模式 / 唯一岗位上限 | 控制关键词扩展范围和跨关键词去重后的总收集量 |
| 硬筛选 | 配置经验、规模、学历、薪资、地区、时间和文本规则等条件 |
| 筛选方案 | 保存并切换完整搜索与筛选配置 |
| 招呼方案 | 配置 AI 开场规则、固定消息和简历图片 |

保存模型配置前可以点击 **测试连接**。该操作只测试模型接口，不会打开 BOSS 页面，也不会执行投递。

### 4. 执行完整流程

1. 保存基础配置、至少一套筛选方案和一套招呼方案。
2. 点击 **开始收集 + AI 筛选**，等待岗位收集、完整 JD 读取、硬筛选和快速 AI 判断完成。
3. 在 **岗位审核** 中查看岗位详情与理由，对推荐岗位逐个批准，或选择后批量批准。
4. 对信息不足的岗位补充确认；需要时生成详细六维评分或人工覆盖 AI 建议。
5. 点击 **预演已批准岗位**，检查、编辑或重新生成 AI 开场，然后逐岗确认预演。
6. 点击 **正式投递 N 个已批准岗位**，再次核对本批岗位名单和招呼方案。
7. 在批次结果中查看成功、失败和未执行岗位；按需只重试失败岗位、处理剩余岗位或开始下一批。

正式投递会操作 BOSS 页面。建议先使用少量岗位熟悉完整流程，并在发送期间保持登录状态，留意页面提示和实时日志。

## 🔐 数据、权限与隐私

- API Key、简历文字、简历图片、筛选方案、招呼方案、审核状态、预演内容、岗位进度和防重复记录保存在浏览器扩展的本地存储中，不会写入代码仓库。
- 岗位匹配、详细评分和招呼语生成会把必要的简历文字、岗位信息及完整 JD 发送给你选择的模型供应商。
- 扩展固定申请 BOSS 页面及内置模型供应商域名权限；自定义模型仅接受 HTTPS 地址，并在需要时单独申请对应域名权限。
- 项目自身不提供中转服务器，也不会主动扫描或导入既有 BOSS 历史会话。
- 清理当前批次不会删除模型配置、简历、筛选方案、招呼方案、岗位进度和已投防重复记录。

在填写真实简历和 API Key 前，请自行了解所选模型供应商的隐私政策和数据处理方式。

## 🛠️ 技术实现

- **浏览器扩展**：Manifest V3，原生 JavaScript，无前端框架、无构建步骤。
- **界面**：浏览器 Side Panel，负责配置、审核、预演、日志和岗位进度。
- **核心编排**：Service Worker 管理收集、筛选、模型调用、批次状态和页面交接。
- **页面操作**：Content Scripts 分别处理岗位搜索/详情页与聊天页。
- **模型接口**：统一的 OpenAI Chat Completions 风格客户端，适配不同服务商的鉴权和参数差异。
- **状态存储**：`chrome.storage.local`，并对旧版本数据进行兼容迁移。

## 🧪 开发与测试

项目没有 npm 依赖或构建步骤。进入扩展目录后可直接使用 Node.js 内置测试运行器：

```bash
cd "JobCopilot · AI"
node --test tests/*.test.js
```

检查所有 JavaScript 文件语法：

```bash
find src tests -type f -name '*.js' -print0 | xargs -0 -n1 node --check
```

当前主开发分支包含 166 项自动化测试，覆盖模型适配、搜索与筛选、人工审核、预演、岗位身份、聊天页交接、失败策略、批次生命周期、存储和扩展集成约束。涉及 BOSS 页面真实 DOM 或真实发送的变化仍应在少量岗位上进行人工验证。
